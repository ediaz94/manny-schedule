## 4. Architecture

**Buildless static PWA — no framework, no build step, no server, no accounts, $0, offline-capable.** Hosted on GitHub Pages; all per-user data in the browser's `localStorage` (key `manny.v1`). Installable (manifest + service worker) so it feels native on iPhone.

**Files:**
- `index.html` — shell; loads scripts in order: `js/foods.js`, `js/data.js`, `js/store.js`, `js/ui.js`, `js/screens-core.js`, `js/screens-plan.js`, `js/app.js`.
- `js/foods.js` — `window.FOODS`: ~1,328 foods (generated by `gen_foods.py`; each `{n,u,cal,p,k[],c,f,col}` = name/unit/cal/protein/keywords/carbs/fat/color).
- `js/data.js` — `window.DATA`: profile defaults, phases (Manny's fitness phases), block-type icons, day templates (Manny's default schedule), workouts (+starting weights, form cues, YouTube demo links), meal templates (w/ macros), **26 dinners** (recipes; first 7 are the rotation, rest are `dow:null` extras), meal-prep tasks, grocery staples, fintech modules/milestones, 38 wedding tasks, monthLabels, vendors, memoriam, `foods` (= window.FOODS w/ inline fallback), `foodSeedVersion` (3).
- `js/store.js` — `window.DateU` (date helpers incl. `fromMin`), `window.Phases` (Manny's phase logic + `daysToWedding`/`eventDate`), `window.Store` (all state in localStorage: load/save, schedule, tracks, day logs, weights, workout logs + drafts, meals/dinnerPlan/weekMeals/grocery, foods consumed + editable foodDb, fintech, wedding tasks + share-sync, vendors, memoriam, mass, reviews, PIN, onboarding). Schedule generator `buildSchedule()` lives here.
- `js/ui.js` — `window.UI`: escaping, toast, bottom-sheet, confirm dialog, SVG charts (weight line, bars, ring), segmented control, nav icons.
- `js/screens-core.js` + `js/screens-plan.js` — `window.Screens`: every screen returns an HTML string. Core = today, workout(+history/progress), weight, meals(+plan/rotation/grocery/prep), faith. Plan = wedding(+vendors/memoriam/busy-map), fintech, review, stats, more, settings, foodList, onboard.
- `js/app.js` — `window.App` (router via hash, bottom nav, PIN gate, phase modal, onboarding gate, share-link detection) + `window.Act` (every `data-act` button handler; global click + change + input delegation).
- `styles.css` — full design system (wedding palette: sage `#2E4A3F`, purple `#8B7298`).
- `manifest.webmanifest`, `sw.js`, `icons/` (from `make_icons.py`), `gen_foods.py`, `README.md`.

**Pattern:** screens render HTML strings with `data-act="..."` attributes; `Act` dispatches them; mutations go through `Store` then `App.render()`. No virtual DOM. Sheets/pickers manipulate their own DOM to avoid full re-renders (preserves focus/typed input).

## 6. Key conventions & gotchas

- **sw cache:** bump `manny-plan-vN` every change (now v23).
- **foodDb migration:** `foodSeedVersion` (now 3); `migrateFoodDb()` adds new seed foods by name AND backfills new fields (e.g. carbs/fat/color) onto existing/edited foods without clobbering user edits.
- **Onboarding gate:** `state.onboarded`; existing users grandfathered to `true` in `load()` so they never see the wizard. `Store.days()` returns `state.schedule || DATA.days` (Manny → default).
- **Wedding seed = 38 tasks** (ids 0–37). Share-sync ships only diffs + custom tasks; merge keeps "done" on either side, dedupes custom by title, never deletes, idempotent.
- **weekMeals** (default `[0..6]`) drives the grocery list and distributes to `dinnerPlan` for "tonight's dinner."
- **Macros estimated** in `gen_foods.py` (carbs/fat split of non-protein calories by keyword; color by keyword/prefix). Re-run `python gen_foods.py` to regenerate `js/foods.js`.
